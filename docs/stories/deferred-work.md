## Deferred from: code review of story-2.1b (2026-07-13)

- **Synchronous Supabase client called inside async LangGraph nodes ‚Äî blocks the event loop under concurrency** ‚Äî every Phase A node (`extract`, `structure`, `chunk`, `embed`) already calls the synchronous `supabase-py` client directly inside an `async def` with no `asyncio.to_thread` wrapping; Story 2-1b's checkpoint helpers (`_read_phase1_checkpoint`, `_write_phase1_checkpoint`) inherit the same pattern. Pre-existing project-wide issue, not introduced by this diff ‚Äî but this is the first place `6√óN` real concurrent `Send()` dispatches make it actually matter (blocking calls serialize what should be parallel I/O). Needs a dedicated async-client migration story, not a one-off fix here.
- **Same-key duplicate-dispatch race ‚Äî the checkpoint read isn't atomic with the write** ‚Äî two concurrent dispatches for the SAME checkpoint key could both observe a cache-miss before either writes, both calling the LLM (duplicate cost, not data loss). Shouldn't occur under the normal crash-then-sequential-retry model this story targets; closing it fully would need a distributed lock (e.g. Redis-based), which is real added scope beyond per-section idempotency.
